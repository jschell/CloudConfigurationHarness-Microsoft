### Blocker: Task 6's unseeded run and Task 7's real A/B run did not execute

Every headless `claude -p ...` invocation in this environment fails with:

    Failed to authenticate. API Error: 401 {"type":"error","error":{"type":"authentication_error","message":"Invalid authentication credentials"}}

`~/.claude/.credentials.json` holds an OAuth token (`subscriptionType:
"pro"`) whose `expiresAt` is already in the past; something about
running `claude -p` as a subprocess from this harness doesn't trigger the
refresh that the interactive session apparently gets. This reproduces on
the *unmodified* smoke test (`_smoke_test.yaml`), confirming it's an
environment/credential issue, not a regression from this session's runner
changes. Options going forward (need the user, not a workaround from
here): re-run `claude` interactively to force a token refresh, or set a
role-specific `*_ANTHROPIC_API_KEY` in `harness/engine/.env` (the runner
already supports this override per role) so `_invoke_claude` uses direct
API-key auth instead of the OAuth session.

Once that's resolved: Task 6 still needs a real unseeded run (`python -m
harness.engine.runner harness/workflows/storage-atomic-tier.yaml`, no
`--context` seed) to see whether `schema_extract` independently proposes
the `networkAcls.defaultAction` hypothesis from `swagger-refs.md`, and
Task 7 still needs `az`/`conftest` installed (or another preflight-driven
decision from the user) before `fixture_validate` can produce a non-stub
verdict for either A/B leg -- and, per this session's earlier decision, a
real Z.ai key in `harness/engine/.env` before the `executor_glm` leg can
run at all (paused, awaiting the user to supply it).
