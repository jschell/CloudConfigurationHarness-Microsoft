## Session 3 (2026-07-01, same machine): both auth/tooling blockers resolved for real

The user installed `conftest` (standalone binary, no admin needed) and
`az` CLI (via an elevated `choco install azure-cli -y`), and supplied
`claude setup-token` credentials in `harness/engine/.env`. Getting from
"tool is present" to "adapter actually produces a correct verdict"
surfaced three more real bugs, all now fixed and verified live (not
stubbed):

1. **OAuth token routed to the wrong env var.** `claude setup-token`
   issues a long-lived token (`sk-ant-oat01-...`) meant for headless use,
   but `_invoke_claude` was putting whatever was in `*_ANTHROPIC_API_KEY`
   into `ANTHROPIC_API_KEY` -- the CLI rejects `sk-ant-oat...` there with
   "Invalid API key"; it needs `CLAUDE_CODE_OAUTH_TOKEN` instead. Fixed by
   detecting the token prefix and routing to the right variable.
   Verified: the smoke test now runs `ping -> pong -> end` for real
   against the live API for both `orchestrator` and `executor_claude`.
2. **`subprocess.run(["az", ...])` fails on Windows.** `az` is a `.cmd`
   shim; Python's subprocess (list-args form, no `shell=True`) doesn't
   apply PATHEXT resolution the way a shell does, so every call raised
   `WinError 2` even with `az` correctly on PATH. Fixed in both
   `bicep_validate.py` and `preflight.py`'s `check_az` by resolving the
   real path via `shutil.which("az")` first.
3. **The one checked-in rule (`AZ-STOR-001.rego`, from Task 5) doesn't
   actually match real `az bicep build` output.** Two separate defects,
   both invisible under Task 5's stub:
   - It used Rego v0 syntax (`deny[msg] { ... }`), which fails to parse
     under the OPA version conftest 0.68.2 bundles (OPA 1.15.2 defaults to
     v1, requiring `if`/`contains`).
   - It checked `input.type`/`input.properties` directly, but
     `az bicep build`'s real output is a full ARM template with a
     top-level `resources` array, not a bare single-resource object --
     so even with valid syntax the rule matched nothing.
   Fixed the rule (`deny contains msg if { some resource in input.resources
   ... }`) and updated `harness/workflows/prompts/rule_compile.md` so
   future `rule_compile` LLM output doesn't repeat either mistake.
   **Verified live**: ran `bicep_validate.bicep_to_json` against both
   real fixtures with real `az bicep build`, then real `conftest` via
   `rego_validate.validate` -- `vulnerable.bicep` -> `fail` (correct),
   `safe.bicep` -> `pass` (correct). Also ran the full
   `handlers.fixture_validate` function against a seeded journal to
   confirm the runner-level wiring, not just the adapters in isolation.

`python -m harness.engine.preflight harness/workflows/storage-atomic-tier.yaml`
now reports `az: OK` and `conftest: OK` on this machine.

## Session 5 (2026-07-01, same machine): journal history tables + a real cross-rule contamination bug

Two follow-ups from Session 4, both driven by the user directly: (1) fix
`compare_runs.py`'s known limitation that it can't diff two runs sharing a
check_id, and (2) a question about whether the harness has the persistent
"what was done and why" record the plan's journal/git split calls for.
Both turned out to be the same underlying gap.

**Schema fix, aligning the implementation with the plan's stated design**
(*"journal tracks provenance and run history, git tracks content
history"*): `rules`/`fixtures` were current-state-only -- one mutable row
per check_id, overwritten in place by whichever workflow_run touched that
check_id last. `runs` was already append-only, but nothing else was. Added
`rule_history`/`fixture_history` (append-only, tagged with
`workflow_run_id`) to `schema.sql`, threaded `workflow_run_id` into
handler `context` via `runner.resume()`, and updated
`handlers.rule_compile`/`fixture_generate` to write to them alongside the
existing current-state tables. `compare_runs.py` now reads rule/fixture
content from history instead of live files or the shared current-state
row, and derives each run's own "validated" outcome from its own
timestamp-scoped `runs` rows instead of trusting the (possibly since-
overwritten) `rules.status` -- this removes the old degraded-diff
fallback (the old "run B overwrote run A's file" message) entirely; it's
no longer needed.

**Bug this surfaced, independent of the schema fix:** re-running a second
A/B pair (hypothesis 4, `minimumTlsVersion`, targeting `AZ-STOR-004`) to
verify the new history tables hit a real, serious pre-existing bug. Every
check's `.rego` file declared `package main`, and
`conftest --policy rules/azure/storage/` loads and evaluates every file in
that directory together -- so *any* check's `deny` rule could fire against
*any other* check's fixtures. Concretely: `AZ-STOR-003`'s rule (from the
first A/B) treats an absent `supportsHttpsTrafficOnly` as risky; testing
`AZ-STOR-004`'s fixtures (which don't set that property at all, since it's
irrelevant to `minimumTlsVersion`) tripped `AZ-STOR-003`'s rule too,
causing spurious `expected=pass actual=fail` results. Checking all 4 rules
against the shared directory confirmed **`AZ-STOR-001` and `AZ-STOR-002`
were both silently broken** by this, despite `rules.status='validated'`
from when each was originally checked (before the later, stricter rules
existed) -- exactly the kind of regression that per-run history (this
session's other change) makes visible instead of invisible. It also
caused `AZ-STOR-004`'s GLM (run B) attempt to be **wrongly rejected** after
3 retries -- re-testing its final rule/fixture output in isolation
(recoverable only because of `rule_history`/`fixture_history`) confirms it
actually passes cleanly; the rejection was a harness false negative, not a
real defect in GLM's output. See the corrected
`docs/task-runs/2026-07-01-az-stor-004-ab-comparison.md` for detail.

**Fix:** each check now gets its own Rego package/conftest namespace
(`checks.<check_id lowercased, dashes to underscores>`, e.g.
`checks.az_stor_004`), enforced via `rego_validate.namespace_for_check_id()`
and `conftest test --namespace <ns>`. Updated all 4 existing `.rego`
files' package declarations and `rule_compile.md` so future generated
rules follow the same convention. Re-verified all 4 checks pass
independently against the shared directory after the fix; restored
`AZ-STOR-004` on disk to the run A (Claude) content, which validated
cleanly both before and after the fix, since run B's rejected-at-the-time
content was what was left on disk after the two runs (see the comparison
doc for why keeping either would have been defensible once the false
negative was understood).

**Not done, flagged as a real gap rather than silently left broken:**
`AZ-STOR-001`/`AZ-STOR-002` are fixed by the namespace change (verified),
but this session didn't add automated regression testing -- there's still
no step that re-checks *all* previously-validated fixtures whenever a new
rule joins the shared directory. The namespace fix prevents the specific
contamination mechanism seen here, but doesn't replace a real regression
gate. Worth a follow-up task.
