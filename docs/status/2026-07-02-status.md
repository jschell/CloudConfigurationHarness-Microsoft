## New reusable patterns (docs/patterns/)

- **schema-coverage-discovery.md** -- deterministically enumerate every
  property on a resource (`harness/tools/enumerate_schema_properties.py`),
  classify each in small batches (`harness/tools/run_schema_coverage.py`),
  track completeness in a `schema_coverage` ledger
  (`harness/tools/coverage_status.py`). Replaces curated doc excerpts +
  LLM judgment alone.
- **rego-rule-authoring.md** -- prefer denying on the negation of a
  property's safe value over matching one named risky value, whenever the
  property has more than two states.
- **deterministic-check-id-assignment.md** -- never let a stateless model
  call invent an identifier whose uniqueness matters; assign it in code.

## Three real bugs found and fixed while actually running this

1. **`AZ-STOR-004` (`minimumTlsVersion`) only checked `== "TLS1_0"`**,
   missing the equally-deprecated `TLS1_1`. Confirmed live with a
   standalone `TLS1_1` fixture before fixing; rewrote to `!= "TLS1_2"`.
   See `docs/patterns/rego-rule-authoring.md`.
2. **check_id collision.** `rule_compile` asked the model to invent its
   own `check_id` with no visibility into which numbers were taken (and
   no memory across stateless calls even with visibility). Three real
   collisions happened during the batch buildout (`AZ-STOR-006`, `-014`,
   `-020`, each assigned to two different hypotheses), each silently
   overwriting an already-validated rule on disk while `rules.
   hypothesis_id` kept pointing at the original. Repaired using
   `rule_history`/`fixture_history` (recoverable only because that
   ledger existed). Root-cause fix: `handlers._check_id_for_hypothesis`
   assigns check_id (and the Rego package name) deterministically in
   code; the model is never asked for either. See
   `docs/patterns/deterministic-check-id-assignment.md`.
3. **`_extract_json` broke on multi-blob model output.** GLM
   occasionally second-guesses itself mid-response (draft JSON, "wait,
   let me reconsider," revised JSON) -- the old first-`{`-to-last-`}`
   fallback spanned both blobs plus the prose between them. Rewrote to
   find every top-level balanced `{...}`/`[...]` span (combined-depth
   tracking so nested objects inside an array aren't mistaken for
   separate top-level spans; string-literal-aware so Rego's own `{`/`}`
   don't miscount) and prefer the last one that parses.

All three are documented with the "why," not just the "what," specifically
so they don't get silently reintroduced by a future session or a new
resource type.

## Known gaps, not actioned (flagged, not fixed)

- **Tier 1 vs Tier 2/3 (atomic vs. combination).** `schema_coverage`
  only ever asks "is this property risky by itself." A property
  correctly marked not-relevant standalone could still be half of a real
  Tier-2 (pattern/combination) finding -- nothing here checks that, and
  the plan's non-goals explicitly scope this phase to Tier 1 only. Every
  property's rationale is preserved regardless of verdict, so a future
  Tier-2 pass could re-scan without re-deriving anything.
- **`AZ-STOR-041` (`sasExpirationPeriod`) is narrower than ideal.** It's a
  duration string (`"365.00:00:00"`), not a small enum -- there's no
  single safe value to negate against (`7.00:00:00` is also fine), so the
  rule only catches the literal 365-day value rather than "too long" in
  general. Would need actual duration parsing/comparison in Rego to fix
  properly; not done.
- **No automated regression gate.** Adding a new rule to `rules/azure/
  storage/` doesn't automatically re-check all previously-validated
  fixtures. The per-check Rego namespace fix (from the prior session)
  prevents the specific cross-contamination mechanism already found, but
  it isn't a general safety net against a differently-shaped bug.
- **`rule_compile.md`'s "prefer `!=` over `==`" guidance is prompt-level,
  not enforced.** `AZ-STOR-041` shipped with `==` despite the guidance
  existing and despite hypothesis 30 (a very similar threshold property,
  compiled moments earlier in the same batch) getting it right --
  confirms this is inconsistently followed, not a one-off.
